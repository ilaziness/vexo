## Context

当前数据恢复流程：
1. 用户选择版本点击恢复
2. 前端调用 DownloadSync
3. 后端下载并解压数据到临时目录
4. 关闭数据库连接
5. 备份原数据目录
6. 移动临时目录到目标位置（覆盖原文件）

问题：第6步会因数据库文件被占用而失败，因为 SQLite 数据库文件在应用运行期间保持打开状态。需要在第5步之后、第6步之前关闭数据库。

## Goals / Non-Goals

**Goals:**
- 解压完成后、替换文件前关闭数据库连接
- 恢复完成后提示用户手动重启应用
- 确保数据文件可以被成功覆盖

**Non-Goals:**
- 自动重启应用（仅提示用户手动重启）
- 在线恢复（不关闭数据库）
- 部分数据恢复（只恢复特定表）
- 提供重启按钮或其他重启机制

## Decisions

### 1. 数据库关闭时机
- **选择**: 在解压完成后、执行文件替换前关闭数据库
- **理由**: 尽可能缩短数据库关闭时间，只在必要时关闭
- **替代方案**: 在 DownloadSync 开始时关闭 - 被拒绝，关闭时间过长影响用户体验

### 2. 重启提示方式
- **选择**: 前端弹框提示用户手动重启应用，不提供自动重启功能
- **理由**: 简化实现，用户自行决定何时重启
- **替代方案**: 提供重启按钮调用应用重启 - 被拒绝，增加复杂度且不必要

### 3. 恢复失败处理
- **选择**: 如果恢复失败，保持数据库关闭状态，提示用户重启
- **理由**: 数据可能已部分覆盖，状态不确定，重启最安全
- **替代方案**: 尝试重新打开原数据库 - 被拒绝，数据一致性无法保证

## Risks / Trade-offs

- [用户体验中断] → 恢复后必须重启，但这是确保数据一致性的必要代价
- [并发安全] → 关闭数据库期间其他服务无法访问数据，需要在UI上禁用相关功能
- [恢复失败回滚] → 如果恢复失败，需要能重新打开原数据库

## Migration Plan

1. 添加数据库关闭接口到 Database 类
2. 修改 Download 流程：解压完成 → 关闭数据库 → 执行替换
3. 前端添加重启提示弹框（仅提示，无操作按钮）
4. 测试恢复成功和失败场景
