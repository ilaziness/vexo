## Context

Vexo AI 命令助手 V2 需要从终端内嵌方案升级为全局侧边栏架构，支持独立的多轮对话管理。现有的 V1 版本没有持久化会话数据，每次重启后对话历史丢失。Phase 1 后端开发需要建立数据持久化层（SQLite）和扩展 AI 服务以支持会话管理和多轮对话。当前后端已有基础的 AIService（基于 Genkit），但缺少会话存储和上下文管理能力。

## Goals / Non-Goals

**Goals:**
- 建立 SQLite 数据库持久化 AI 会话和消息数据
- 实现会话 CRUD 操作接口（创建、查询、列表、删除）
- 扩展 AIService 支持多轮对话（传递历史上下文）
- 支持停止生成功能
- 生成前端 TypeScript 绑定

**Non-Goals:**
- 前端 UI 实现（侧边栏、聊天界面等）
- 终端联动功能（选中文本发送到 AI）
- V1 数据迁移（V2 为全新开发，不兼容 V1）
- AI 模型切换、配置管理等高级功能

## Decisions

**数据库迁移：复用现有 SQLite**
- 理由：项目已使用 SQLite 数据库，无需新增依赖，只需添加表结构迁移
- 替代方案考虑：新建独立数据库文件（增加复杂度、管理成本）

**数据访问层设计：Repository 模式**
- 理由：解耦数据库实现与业务逻辑，便于测试和未来迁移
- 替代方案考虑：直接在 Service 层操作数据库（耦合度高、难以测试）

**Genkit Flow 扩展：复用现有 chatFlow**
- 理由：V1 已有基础的 chatFlow，扩展支持多轮对话上下文传递即可
- 替代方案考虑：新建专用 multiTurnChatFlow（代码重复，维护成本高）

**会话标题生成：前端自动生成**
- 理由：减少后端复杂度，前端更灵活（首条消息前 20 字）
- 替代方案考虑：后端 AI 生成标题（增加 API 调用成本、延迟）

**停止生成实现：context 取消**
- 理由：Go 标准模式，Genkit 支持取消上下文
- 替代方案考虑：共享标志位（不优雅、无法立即停止）

## Risks / Trade-offs

**[风险] SQLite 并发写入冲突** → 使用事务和适当的锁机制，桌面应用并发压力较小
**[风险] Genkit 停止生成不支持** → 验证 Genkit 框架是否支持 context 取消，如不支持则需标记为待实现
**[权衡] 数据库迁移复杂度** → Phase 1 仅创建新表，暂不处理迁移逻辑（V2 为全新开发）
**[权衡] 前端绑定频繁生成** → 仅在 Phase 1 最后生成一次，避免频繁重新生成

## Migration Plan

1. **数据库迁移**：使用现有的 migration.go 机制在现有 SQLite 数据库中添加 ai_sessions、ai_messages 表
2. **Schema 迁移**：在应用启动时检查并创建新表（如不存在）
3. **前端绑定生成**：完成后端开发后运行 `wails3 generate bindings -ts`
4. **回滚策略**：如需回滚，删除新增的数据库表和代码即可（无数据迁移需求）
