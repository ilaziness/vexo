## Context

当前同步功能已实现版本上传、下载和列表查看。服务端限制每个用户最多保留5个历史版本，客户端在历史版本列表中只能查看和恢复，无法删除特定版本。

## Goals / Non-Goals

**Goals:**
- 在客户端历史版本列表每项后添加删除按钮
- 删除前显示确认弹框
- 同步设置页面历史记录列表只展示前10个版本
- 恢复数据弹框内历史记录支持滚动加载，每次加载50条
- 将服务端版本存储上限从5个增加到500个
- 服务端添加删除指定版本的API端点

**Non-Goals:**
- 批量删除功能
- 版本回收站/恢复已删除版本（删除是永久性的，无备份）
- 版本命名/备注功能

## Decisions

### 1. 删除确认交互方式
- **选择**: 使用 MUI Dialog 组件显示确认弹框
- **理由**: 与现有恢复数据弹框风格一致，保持UI统一性
- **替代方案**: 使用浏览器原生 confirm() - 被拒绝，因为风格不统一

### 2. 服务端版本上限值
- **选择**: 500个版本
- **理由**: 满足用户长期历史保留需求，同时控制存储成本
- **替代方案**: 1000个 - 被认为存储成本过高；100个 - 可能不足以满足长期需求

### 3. 删除API设计
- **选择**: DELETE /versions?version={number}
- **理由**: RESTful风格，使用查询参数传递版本号
- **替代方案**: POST /versions/delete - 被拒绝，不符合RESTful规范

### 4. 版本列表分页策略
- **选择**: 设置页面固定显示前10个版本，恢复弹框使用无限滚动每次加载50条
- **理由**: 设置页面作为预览不需要显示全部；恢复弹框需要浏览所有历史，无限滚动平衡性能和用户体验
- **替代方案**: 统一分页组件 - 被拒绝，两种场景使用模式不同

## Risks / Trade-offs

- [存储成本增加] → 版本上限从5增加到500，存储需求可能增加100倍
- [误删风险] → 添加确认弹框降低误删概率，但无法完全避免
- [API安全性] → 删除操作需要严格的身份验证，防止未授权删除

## Migration Plan

1. 服务端配置变更：修改默认max_versions从5到500
2. 服务端添加分页查询支持（limit/offset参数）
3. 服务端添加删除端点
4. 客户端添加删除按钮和确认弹框
5. 客户端设置页面限制显示前10个版本
6. 客户端恢复弹框实现无限滚动加载（每次50条）
7. 客户端添加删除API调用
8. 测试验证
